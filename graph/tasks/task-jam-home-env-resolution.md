Define `JAM_HOME` env var that defaults to `/home/maestro/.jam` when running as maestro; resolves to `~/.jam` when running as caleb (for CLI). Tools and services use `JAM_HOME` everywhere instead of hardcoded `~/`.

Per security-setup §7.1 path defaults table.

Add a config:
```toml
# /home/maestro/.jam/config/skills.toml
skills-repo = "/home/caleb/code/jam-skills"
```

The Maestro reads this; the inotify watcher uses this path; CLI tools that read skills use this path.

Implementation note (2026-05-06): landed shared path resolution in `crates/jam-tools-core/src/paths.rs` and Python parity in `maestro/src/jam_maestro/paths.py`. `JAM_HOME` wins when set; without it, `maestro` resolves to `/home/maestro/.jam`, while Caleb/the CLI resolves to `$HOME/.jam`. Rust call sites now use the shared resolver in `jam-cli`, `jam-setup`, `jam-nats-bridge`, `jam-ui-server`, `jam-journal-reconciler`, `jam-svc-observe`, `jam-svc-knowledge`, `jam-svc-session`, `jam-pr-poller`, `jam-trunk-fetcher`, and `jam-tempyr-pr-reconciler`. Python Maestro config readers use the same rule for skills, input-budget, and project config defaults.

The current decided skills config shape is the direct-read `[skills] folders/files` schema from `dec-skills-direct-read-with-config`, generated by `scripts/bootstrap-users.sh` at `/home/maestro/.jam/config/skills.toml`; this supersedes the older top-level `skills-repo = ...` sketch above. The Maestro loader and `jam-svc-knowledge` inotify watcher both read that same config path by default.
